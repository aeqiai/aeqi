# aeqi design language

> **v4 — Graphite + Ink.** The aeqi design system is built on restraint.
> Every element earns its place through function, not decoration. The interface
> recedes so the four primitives — Agents, Events, Quests, Ideas — can speak clearly.

> Every value in the tables below is read live from `:root` via `getComputedStyle`.
> There is no hardcoded hex on this page — edit `@aeqi/tokens` or `primitives.css`
> and this doc updates automatically.

---

## Principles

### Restraint, with one accent

Pure neutral paper, ink at opacity, one near-black graphite accent. The accent
earns its weight by scarcity: wordmark, primary CTA, links, focus, active state.
Titles stay ink. The system is restrained, not monochrome.

**Ratio rule — hold these or it goes generic:**

- ~70% neutral grey paper + white card surfaces
- ~28% ink body copy (black at 22–92% alpha)
- ~2% near-black graphite accent (CTAs, active nav, focus, selected, wordmark)

Jade is reserved for the **success** semantic (completed quests, resolved
events) so brand and status signal read visually distinct.

### Precision over decoration

No gratuitous gradients, heavy shadows, or extravagant rounding. Components use
sharp geometry and minimal chrome. Status color (success, error, warning, info)
appears sparingly — a dot, a tint — never a full-bleed wash.

### The four primitives

Everything in aeqi maps to one of four primitives:

- **Agents** — autonomous entities with parent-child hierarchy (WHO)
- **Events** — triggers and audit stream (WHEN)
- **Quests** — work items agents pursue (WHAT)
- **Ideas** — knowledge, identity, instructions, memories (HOW)

## The æqi wordmark

The **æqi** wordmark, set in Zen Dots and rendered in graphite
(`var(--color-accent)`), is the brand mark. It appears as the product name and
never as decoration. Brand primitives (`<Wordmark>`, `<BrandMark>`) default to
the accent — never pass a `color` override unless the surface demands it.

Usage rules:

- Always lowercase: **æqi**, never "AEQI" or "Aeqi" in UI text
- The "æ" ligature is reserved for the logo and brand references
- Never add gradients, shadows, or effects; never rotate or stretch
- The accent owns the wordmark — nothing else competes

## Row rhythm

The app uses a single row height, `--input-h`, for every dense interactive
surface: sidebar nav rows, inputs, buttons (`md` size), the scope indicator.
This is also `--sidebar-row-h`. Buttons and inputs placed on the same row line
up without tweaks. The composer is taller (multi-line) but borrows the same
border/radius/colors so it reads as a member of the same family.

<TokenTable
  rows={[
    { token: "--input-h", usage: "Canonical row height for dense interactive surfaces (32px)" },
    { token: "--sidebar-row-h", usage: "Sidebar nav rows (aliases --input-h)" },
  ]}
/>

Do not introduce new row heights. If a surface needs to feel denser, use `sm`.
If it needs to feel weightier (hero, auth), use `lg`.

## Using the primitives

**Rule: reach for `components/ui/` first.** A new ad-hoc class is the last
resort, not the default. If the thing you want doesn't exist as a primitive,
propose adding it to the library before writing a one-off class in a page-level
stylesheet.

### Which primitive to use when

| Need                                  | Primitive                   | Don't write                                                |
| ------------------------------------- | --------------------------- | ---------------------------------------------------------- |
| Any `<button>`                        | `<Button variant size>`     | `.btn`, `.auth-btn-*`, `.modal-btn-*`, `.newBtn`           |
| Icon-only button                      | `<IconButton variant size>` | Bare `<button>` with `aria-label` + inline SVG             |
| Any single-line text field            | `<Input size>`              | `.auth-input`, `.filter-input`, `.agent-settings-input`    |
| Any multi-line text field             | `<Textarea>`                | `.modal-textarea`, bespoke `<textarea>` rules              |
| Status / tag chip                     | `<Badge>`                   | Hand-rolled status dot + span                              |
| Tag list with removal                 | `<TagList>`                 | Custom chip rows with close-X handlers                     |
| Bordered container (agent, event row) | `<Card variant padding>`    | `.agent-card`, `.event-row`, `.tool-card`, `.welcome-card` |
| Detail panel with title bar           | `<Panel title actions>`     | Hand-rolled `.detail-panel` + `<h2>` header                |
| Detail field (label + value)          | `<DetailField label>`       | Hand-rolled `.field-label` + `<p>` pair                    |
| Hero metrics row                      | `<HeroStats>`               | Custom flex with inline big-number styles                  |
| Empty list / no-results state         | `<EmptyState>`              | Hand-rolled "No results" div                               |
| Loading + error + empty tri-state     | `<DataState>`               | Three nested ternaries in the page                         |
| Error boundary                        | `<ErrorBoundary>`           | Bespoke class component with state                         |
| Progress indicator (determinate)      | `<ProgressBar>`             | Inline `<div style="width: X%">`                           |
| Loading indicator (indeterminate)     | `<Spinner>`                 | CSS-only spinners in page stylesheets                      |
| Tab bar                               | `<Tabs>`                    | Role=tablist handwritten, focus management bespoke         |
| Hover explainer                       | `<Tooltip>`                 | `title` attribute (inaccessible, ugly on mobile)           |
| Modal dialog                          | `<Modal>`                   | `position: fixed` divs                                     |

### Token rules

- `--input-*` tokens are the source of truth for **every field-like surface**.
  `<Input>`, `<Textarea>`, the composer, and the few remaining class-based
  inputs (`.filter-input`, `.agent-settings-input`, `.events-link-input`) all
  consume them. Edit once, update everywhere.
- Never hardcode `#ffffff`, `rgba(0, 0, 0, 0.x)`, or hex colors in component
  CSS. Use a token. If the token you need doesn't exist, add it to
  `@aeqi/tokens` (shared) first, then alias it in `primitives.css`.
- Never hardcode radius values. Use `--radius-md` for inputs/buttons,
  `--radius-lg` for cards/panels, `--radius-xl` for modals.
- Never hardcode transition timings. Use `--transition-fast` for interaction,
  `--transition-normal` for layout.

### Adding a new primitive

1. Start from an existing one (`Input.tsx` / `Input.module.css` are the cleanest template).
2. Props: `variant`, `size`, `className`, then role-specific props. Always accept `className` and forward it last so call sites can extend.
3. CSS module: use `--input-*`, `--space-*`, `--radius-*`, `--text-*` tokens only.
4. Add a Storybook story showing every variant and every size.
5. Export from `index.ts`.
6. Document it in the table above.

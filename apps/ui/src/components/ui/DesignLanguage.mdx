# aeqi design language

The aeqi design system is built on restraint. Every element earns its place through function, not decoration. The interface recedes so the four primitives -- Agents, Events, Quests, Ideas -- can speak clearly.

---

## Principles

### Restraint, with one accent

Warm parchment paper, ink text at opacity, one accent — heraldic gold. The accent earns its weight by scarcity: wordmark, primary CTA, links, focus. Titles stay ink. The system is restrained, not monochrome.

### Opacity as depth

A single black at varying opacities carries hierarchy inside the ink system:

| Token                    | Value                  | Usage                      |
| ------------------------ | ---------------------- | -------------------------- |
| `--color-text-primary`   | `rgba(0, 0, 0, 0.85)`  | Body text, headings        |
| `--color-text-secondary` | `rgba(0, 0, 0, 0.5)`   | Labels, descriptions       |
| `--color-text-muted`     | `rgba(0, 0, 0, 0.3)`   | Hints, timestamps          |
| `--color-border`         | `rgba(0, 0, 0, 0.08)`  | Panel borders, dividers    |
| `--color-bg-surface`     | `rgba(0, 0, 0, 0.025)` | Subtle surface distinction |
| `--color-bg-elevated`    | `rgba(0, 0, 0, 0.05)`  | Hover, elevated panels     |

### Surfaces — paper and card

The canvas is warm parchment; raised surfaces read as "same material, lifted":

| Token           | Value     | Usage                                       |
| --------------- | --------- | ------------------------------------------- |
| `--color-paper` | `#faf6ec` | Page base — the default canvas              |
| `--color-card`  | `#ffffff` | Raised card / panel (landing)               |
| `--bg-elevated` | paper     | Dashboard canvas — consumes `--color-paper` |
| `--bg-base`     | `#fcfaf3` | Dashboard card — warm white on parchment    |

### Precision over decoration

No gratuitous gradients, heavy shadows, or extravagant rounding. Components use sharp geometry and minimal chrome. Status color (success, error, warning, info) appears sparingly — a dot, a tint — never a full-bleed wash.

### The four primitives

Everything in aeqi maps to one of four primitives:

- **A** -- Agents: autonomous entities with parent-child hierarchy
- **E** -- Events: the audit/activity stream
- **Q** -- Quests: work items agents pursue
- **I** -- Ideas: knowledge, identity, instructions, memories

---

## The æqi wordmark

The **æqi** wordmark, set in Zen Dots and rendered in heraldic gold (`var(--color-accent)`), is the brand mark. It appears as the product name and never as decoration. Brand primitives (`<Wordmark>`, `<BrandMark>`) default to the accent — never pass a `color` override unless the surface demands it.

Usage rules:

- Always lowercase: **æqi**, never "AEQI" or "Aeqi" in UI text
- The ligature glyph is reserved for the logo and brand references
- Never add gradients, shadows, or effects; never rotate or stretch
- The accent owns the wordmark — nothing else competes

## Typography

Each typeface has a single clear role. Do not introduce more.

| Typeface       | Variable         | Usage                                     |
| -------------- | ---------------- | ----------------------------------------- |
| Inter          | `--font-sans`    | UI chrome — labels, body, headings        |
| JetBrains Mono | `--font-mono`    | Code, IDs, model names, technical values  |
| Cinzel         | `--font-display` | Page titles, hero, marquee moments        |
| Zen Dots       | `--font-brand`   | The æqi wordmark only                     |
| EB Garamond    | `--font-serif`   | Long-form prose (docs, blog, essay tones) |

### Type scale

| Token              | Size             | Usage                        |
| ------------------ | ---------------- | ---------------------------- |
| `--font-size-3xs`  | 0.5625rem (9px)  | Micro labels                 |
| `--font-size-2xs`  | 0.625rem (10px)  | Eyebrow caps                 |
| `--font-size-xs`   | 0.75rem (12px)   | Badges, timestamps, metadata |
| `--font-size-sm`   | 0.8125rem (13px) | Secondary text, descriptions |
| `--font-size-base` | 0.875rem (14px)  | Body text, form labels       |
| `--font-size-lg`   | 1rem (16px)      | Base / root size             |
| `--font-size-xl`   | 1.25rem (20px)   | Section headings             |
| `--font-size-2xl`  | 1.5rem (24px)    | Page titles                  |
| `--font-size-3xl`  | 1.875rem (30px)  | Hero headlines               |
| `--font-size-4xl`  | 2.25rem (36px)   | Display                      |

Weights: 400 (regular), 500 (medium for labels), 600 (semibold for headings), 700 (bold, rare).

## Status colors

Status colors are the only chromatic exception outside the accent. They appear minimally — badge dots, tinted backgrounds, subtle text colors — never full-bleed washes:

| Status  | Color     | Token             | Used for                            |
| ------- | --------- | ----------------- | ----------------------------------- |
| Success | `#22c55e` | `--color-success` | Done, active, healthy               |
| Error   | `#ef4444` | `--color-error`   | Failed, validation errors           |
| Warning | `#f59e0b` | `--color-warning` | Blocked, degraded                   |
| Info    | `#3b82f6` | `--color-info`    | In progress, informational          |
| Accent  | `#D4A017` | `--color-accent`  | Wordmark, primary CTA, links, focus |

## Row rhythm (32px)

The app uses a single row height, `--input-h: 32px`, for every dense interactive surface: sidebar nav rows, inputs, buttons (`md` size), the scope indicator. This is also `--sidebar-row-h`. Buttons and inputs placed on the same row line up without tweaks. The composer is taller (multi-line) but borrows the same border/radius/colors so it reads as a member of the same family.

Do not introduce new row heights. If a surface needs to feel denser, use `sm` (28px). If it needs to feel weightier (hero, auth), use `lg` (40px).

## Using the primitives

**Rule: reach for `components/ui/` first.** A new ad-hoc class is the last resort, not the default. If the thing you want doesn't exist as a primitive, propose adding it to the library before writing a one-off class in a page-level stylesheet.

### Which primitive to use when

| Need                                  | Primitive                   | Don't write                                                |
| ------------------------------------- | --------------------------- | ---------------------------------------------------------- |
| Any `<button>`                        | `<Button variant size>`     | `.btn`, `.auth-btn-*`, `.modal-btn-*`, `.newBtn`           |
| Any single-line text field            | `<Input size>`              | `.auth-input`, `.filter-input`, `.agent-settings-input`    |
| Any multi-line text field             | `<Textarea>`                | `.modal-textarea`, bespoke `<textarea>` rules              |
| Status / tag chip                     | `<Badge>` / `<StatusBadge>` | Hand-rolled status dot + span                              |
| Bordered container (agent, event row) | `<Card variant padding>`    | `.agent-card`, `.event-row`, `.tool-card`, `.welcome-card` |
| Detail panel with title bar           | `<Panel title actions>`     | Hand-rolled `.detail-panel` + `<h2>` header                |
| Empty list / no-results state         | `<EmptyState>`              | Hand-rolled "No results" div                               |
| Loading + error + empty tri-state     | `<DataState>`               | Three nested ternaries in the page                         |
| Hover explainer                       | `<Tooltip>`                 | `title` attribute (inaccessible, ugly on mobile)           |
| Modal dialog                          | `<Modal>`                   | `position: fixed` divs                                     |

### Token rules

- `--input-*` tokens are the source of truth for **every field-like surface**. `<Input>`, `<Textarea>`, the composer, and the few remaining class-based inputs (`.filter-input`, `.agent-settings-input`, `.events-link-input`) all consume them. Edit once, update everywhere.
- Never hardcode `#ffffff`, `rgba(0, 0, 0, 0.x)`, or hex colors in component CSS. Use a token. If the token you need doesn't exist, add it to `@aeqi/tokens` (shared) first, then alias it in `primitives.css`.
- Never hardcode radius values. `--radius-md` (8px) for inputs/buttons, `--radius-lg` (12px) for cards/panels, `--radius-xl` (16px) for modals.

### Adding a new primitive

1. Start from an existing one (`Input.tsx` / `Input.module.css` are the cleanest template).
2. Props: `variant`, `size`, `className`, then role-specific props. Always accept `className` and forward it last so call sites can extend.
3. CSS module: use `--input-*`, `--space-*`, `--radius-*`, `--text-*` tokens only.
4. Add a Storybook story showing every variant and every size.
5. Export from `index.ts`.
6. Document it in the table above.
